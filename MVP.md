# Product Requirements Document
## VibePM — 面向个人开发者的轻量创意驾驶舱

**版本:** v0.1 (MVP)
**作者:** Jacoy
**日期:** 2026年3月
**状态:** 草稿

---

## 1. 背景与问题陈述

### 背景

Vibe coding 的兴起正在催生一批全新的个人开发者——他们借助 Cursor、Lovable、v0 等 AI 编程工具快速将创意落地，同时并行推进多个项目。这批用户不同于传统开发者：他们的技术背景参差不齐，对 Git、CLI 的熟悉程度有限，但创意密度极高、项目迭代节奏极快。

### 核心问题

> 当一个人同时有 3-5 个进行中的项目，又没有系统化管理习惯时，他的创意和进度极易失控。

现有工具无法很好地服务这批用户：

| 工具 | 问题 |
|------|------|
| Linear / Jira | 学习成本高，功能过剩，工程感过强 |
| GitHub Issues | 需要熟悉 Git 工作流，不够直觉 |
| Notion | 太自由，没有结构，难以形成习惯 |
| 脑子记 | 项目一多立刻失控 |

### 机会

为 Vibe Coder 打造一款**极低摩擦、随时可见、以创意为驱动**的项目管理工具。

---

## 2. 目标用户

### 主要用户：Vibe Coder

- 使用 AI 编程工具（Cursor、Lovable、v0 等）的个人开发者
- 同时推进 2-5 个独立项目
- 创意驱动，缺乏系统化工程习惯
- 主力设备为 MacBook
- 对复杂工具有明显抵触情绪

### 用户核心诉求

1. **快速捕捉** — 灵感出现时，能在 10 秒内记录下来
2. **随时可见** — 不需要切换窗口就能看到所有项目进度
3. **低维护成本** — 不需要花时间"管理工具本身"

---

## 3. 产品定位

**VibePM 是一个为 Vibe Coder 设计的创意驾驶舱。**

它不是项目管理工具，而是帮助创意型开发者在多个项目之间保持清醒、不失控的副驾驶。

### 一句话描述

> "说出你的想法，看着它变成现实。"

---

## 4. MVP 功能范围

### 4.1 便签系统（Task Stamps）

**功能描述**

用户可以自定义可复用的任务便签。每张便签代表一个任务节点，例如："完成登录系统"、"部署上线"、"完成 UI 设计"。

**核心交互**

- 创建项目时，用户用便签拼接出项目的工作流（任务列表）
- 便签可跨项目复用
- 便签支持自定义名称和颜色标识

**设计原则**

- 创建一张便签不超过 5 秒
- 拼接工作流的过程要有"搭积木"的直觉感

**不包含**

- AI 自动生成任务（放入 v2 规划）
- 子任务嵌套
- 截止日期与提醒

---

### 4.2 Menu Bar 全局视图

**功能描述**

产品以 macOS Menu Bar App 的形式存在。用户点击顶栏图标，下拉面板展示所有项目的实时进度快照。

**核心交互**

- 面板展示所有项目名称 + 进度百分比 + 当前进行中的任务
- 点击项目可展开查看完整便签列表
- 可直接在面板内进行 Commit 操作（见 4.3）

**设计原则**

- 打开面板到获取信息不超过 3 秒
- 视觉设计要给人"一切尽在掌控"的感受
- 面板收起后不打扰用户

---

### 4.3 Commit 交互

**功能描述**

用户完成某项任务后，可在 Menu Bar 面板内直接将对应便签标记为完成，记录一次"commit"。

**核心交互**

- 拖动便签 或 点击 ✅ 表示该任务完成
- 完成后项目进度自动更新
- 支持为每次 commit 附加文字备注（不限字数，给喜欢写长描述的用户充分发挥的空间）

**设计原则**

- Commit 一次不超过 3 步操作
- 文字备注是可选的，不是必填的

**不包含（放入 v2）**

- Git 自动同步
- Commit 历史时间线视图

---

## 5. 不在 MVP 范围内的功能

| 功能 | 理由 | 规划版本 |
|------|------|---------|
| AI 任务拆解 | 需验证用户是否愿意手动建立工作流 | v2 |
| Git 自动同步 | 开发复杂度高 | v2 |
| 多人协作 | 违背个人开发者定位 | 暂不规划 |
| 移动端 | 先把 Mac 体验做到极致 | v3 |
| Commit 历史时间线 | MVP 阶段专注进度感知 | v2 |

---

## 6. 核心验证假设

MVP 阶段最重要的问题只有一个：

> **Vibe Coder 是否愿意自发地建立自己的便签工作流，并且持续复用它？**

### 判断标准

| 指标 | 健康信号 | 警示信号 |
|------|---------|---------|
| 便签创建数 | 用户平均创建 5+ 张便签 | 用户只创建 1-2 张后停止 |
| 复用率 | 30% 以上的便签被跨项目复用 | 每个项目都从零开始建便签 |
| 7日留存 | 用户在第 7 天仍有 commit 行为 | 用户在第 3 天后停止使用 |

### 如果假设成立

便签心智模型被验证 → v2 引入 AI 自动生成便签，将"手动搭建工作流"升级为"描述项目，AI 帮你拆解"，体验质的飞跃。

### 如果假设不成立

提前暴露问题 → 调整方向，避免在 AI 功能上投入大量资源后才发现用户根本不需要结构化工作流。

## 7. 用户故事

**场景一：新项目启动**

> 小明有了一个新想法：做一个 AI 简历生成器。他打开 VibePM，输入项目名称，然后从自己的便签库里拖出几张常用便签组成工作流：「搭建基础框架」→「完成 UI」→「接入 AI API」→「测试上线」。30 秒内，一个清晰的项目计划出现在眼前。

**场景二：日常推进**

> 小明正在用 Cursor 写代码，突然想看看今天还有什么没做。他不需要切换窗口，只是瞥了一眼 Menu Bar 图标，点击一下，看到「AI API 接入」这张便签还没完成。他继续写代码。

**场景三：记录进度**

> 小明完成了用户注册系统，他在 Menu Bar 面板里把对应便签拖到完成区域，顺手写了一段备注："花了比预期多 2 倍的时间，主要卡在 OAuth 回调上，下次优先看文档。" 项目进度从 40% 跳到 60%。

## 8. 成功指标

| 指标 | MVP 目标 |
|------|---------|
| 激活率 | 注册后 24 小时内完成第一个 commit |
| 7日留存率 | ≥ 40% |
| 人均项目数 | ≥ 2 个 |
| 便签复用率 | ≥ 30% |
| NPS | ≥ 50 |

---

## 9. 技术架构方向（初步）

- **平台:** macOS Menu Bar App（优先）
- **框架:** Electron 或原生 SwiftUI（待定）
- **数据存储:** 本地优先，iCloud 同步（MVP 阶段可纯本地）
- **AI 集成:** 暂不引入（v2 规划）

---

## 10. 版本规划

| 版本 | 核心功能 | 目标 |
|------|---------|------|
| v1 (MVP) | 便签系统 + Menu Bar + Commit 交互 | 验证核心假设 |
| v2 | AI 任务拆解 + Git 自动同步 | 消除摩擦力 |
| v3 | Commit 历史时间线 + 数据洞察 | 提升留存 |
